## Key Decisions

- **No framework (vanilla JS + plain CSS)**: chosen for small bundle size and simplicity. Rejected: React/Vue/Svelte (overkill for a small tuner), Tailwind (build overhead not worth <5KB of styles).
- **pitchy for pitch detection**: McLeod Pitch Method, small (~2KB), actively maintained. Rejected: pitchfinder (multiple algorithms but larger), custom FFT (reinvent the wheel).
- **Singleton AudioContext with lazy init**: lets consumers call `getAudioContext()` freely but `initAudio()` is explicit for iOS gesture compliance.
- **CustomEvent dispatch for cross-module communication**: `play-tone`, `chromatic-mode-change`, `reference-pitch-change` on document. Rejected: shared state object / event emitter class (overkill for ~5 events).
- **SVG gauge over Canvas**: SVG scales cleanly, DOM-integrated, smaller bundle. No need for Canvas's imperative drawing for a simple needle.
- **Colorblind-safe palette** (teal/amber/orange + arrows), not red/green.
- **In-tune threshold ±10 cents**, direction dead-zone ±5 cents.
- **`style.transform = rotate(Xdeg)` not `setAttribute('transform', ...)`**: CSS transitions don't respect SVG's inline rotation center. (Discovered via Bug 3.)

## Patterns & Discoveries

- **`exponentialRampToValueAtTime` cannot target 0** — use 0.0001 as floor, and call `cancelScheduledValues` + `setValueAtTime(gain.value, now)` before new ramps mid-envelope.
- **AnalyserNode must NOT connect to `ctx.destination`** — source → analyser only, otherwise mic feedback loop.
- **OscillatorNode is one-shot** — create fresh instance per tone play.
- **iOS Safari** starts AudioContext suspended — must call `resume()` after user gesture.
- **Vite serves files in `public/` at root** — service worker lives there (not in src/) for proper scope.
- **Ukulele uses re-entrant tuning** (G4 higher than C4) — don't assume strings ordered low-to-high.
- **Pre-commit hooks in this repo may batch-commit staged changes from multiple agents** into a single commit. Verify post-commit that changes landed.
- **Root cause of 3 post-sprint bugs:** Component-level sprint dispatch with no explicit integration wiring task. main.js was an afterthought with 6 churn events.
- **"Build passes" is not verification.** CSS/SVG/audio bugs only surface at runtime in a browser.

## Mental Model

The app has 4 clean layers:
1. **Audio layer** (`src/audio/`): AudioContext → microphone → pitch detector → note detector → tone generator. All modules are singletons with module-scoped state.
2. **Data layer** (`src/data/`, `src/utils/`): Pure instrument data + music math. All functions accept optional `a4` param for configurable reference pitch.
3. **UI layer** (`src/ui/`, `src/styles/`): Vanilla DOM components that mount into predefined containers in index.html. Components communicate via `document.addEventListener` on CustomEvents.
4. **Orchestration** (`src/main.js`): The integration layer. Imports all modules, mounts UI components, wires pitch detection loop to gauge updates, bridges CustomEvents between modules.

Current runtime flow (after bug fixes):
```
User clicks "Start Tuning" → permission-flow → microphone.startMicrophone() → AnalyserNode
  → pitch-detector.startPitchDetection(analyser, callback)
    → per frame: detectNote(frequency, strings, a4) → gauge.update(...)
                                                   → setActiveString(index)
```

## Open Questions

1. **Icon assets**: `public/icon-192.svg` and `public/icon-512.svg` are placeholder tuning-fork SVGs. Android Add-to-Home-Screen prefers rasterized PNGs. Options: (a) ship with SVG for now — works on iOS and Chrome desktop (b) generate proper 192/512 PNGs. Criteria: target platforms for v1. Ask: user/design.

2. **Opus model override validation**: The retro identified that integration and complex UI/SVG tasks should use opus instead of sonnet. This is documented in team.yaml but not tested yet. Next feature sprint should try dispatching an integration task with `model: opus` override and compare quality. If validated → promote to rule.

3. **Note-detector direction threshold inconsistency**: In-tune threshold is ±10 cents, but direction dead-zone is ±5 cents. Between 5-10 cents, `inTune: false` but `direction: 'in-tune'`. Is this intentional? May confuse downstream UI logic. Ask: user for preference, or pick one threshold.

4. **Reference tone during mic capture**: Playing a reference tone while the mic is active could cause feedback. Current code requests `echoCancellation: true` in getUserMedia which should handle most cases, but not tested with mic + tone simultaneously on low-end Android speakers. Criteria: real-device testing. Ask: defer until someone tests on Android.

## Risks & Warnings

- **Bug 3 regression risk**: The SVG transform-origin fix depends on CSS `transform-origin: 200px 180px` matching the viewBox CX/CY constants in `tuner-gauge.js`. If someone changes the gauge viewBox dimensions in JS, they must also update the CSS. Consider adding a comment in gauge.css referencing the JS constants.
- **main.js is fragile**: 6 commits touched main.js during the bug-fix phase. It's the integration layer and easy to break. Any new feature that touches multiple modules should update main.js explicitly as a task.
- **iOS audio unlock**: The app relies on `initAudio()` being called from a user gesture (the Start Tuning click). If any code path tries to play audio before that gesture, iOS will silently fail. The permission-flow module enforces this today, but future code must route through it.
- **Service worker caching**: Dev iteration can be confusing if the SW caches an old bundle. During development, clear site data or unregister SW in DevTools after major changes.
- **52 unit tests all pass** but only cover `music.js` and `note-detector.js` (pure functions). Audio modules, UI modules, and integration have zero test coverage — all verification is manual.
